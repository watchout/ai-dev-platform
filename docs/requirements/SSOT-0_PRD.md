## 1. プロダクト概要

### 1.1 一言で何か（エレベーターピッチ）

```
個人開発者・スタートアップチーム・非エンジニア 向けの AI開発フレームワーク で、
AI開発が規模拡大とともに破綻する問題 を解決する。
Cursor/Lovable等のバイブコーディングツール とは異なり、
バイブコーディングの手軽さを維持しつつ裏側でSSOT駆動開発を自動化する一気通貫フレームワーク を提供する。
```

### 1.2 背景・課題

現状の課題を具体的に記述：

- AIでの開発は小規模なら高速だが、規模が大きくなるとコードが破綻し完成しない
- 開発経験者でもAI活用は部分的な工数削減にしかならず、期待ほどスピードが上がらない
- 個人開発者はAIへの指示出しに時間とコストをかけても一向に完成しない
- 非エンジニアはAI開発に挑戦しても結局知識が必要になり諦めてしまう
- AIは同じパターンのミスを繰り返し、修正コストが開発コストを上回る

### 1.3 解決策

本プロダクトがどのように課題を解決するか：

- SSOT（Single Source of Truth）仕様書を自動生成し、AIが迷わないガードレールを整備する
- ヒアリングベースの対話フローで開発者の意図を最大限に仕様書に反映する
- 仕様書からタスク分解・プロンプト生成・実装・監査を自動化する
- リポジトリベースの自動開発により、PC環境に依存しない開発を実現する
- 判断が必要な場面では即座にユーザーに確認し、推測で進めない

## 2. 対象ユーザー

### 2.1 プライマリユーザー: 個人開発者

| 項目 | 内容 |
|------|------|
| ペルソナ名 | 個人開発者 |
| 役割 | プロダクトオーナー兼開発者 |
| 課題 | AI開発で規模が大きくなると破綻する。部分的なAI活用では効率が限定的 |
| ゴール | アイデアから本格的なプロダクトを完成させる |
| 利用頻度 | 日常的（毎日） |

### 2.2 セカンダリユーザー: スタートアップエンジニアチーム

| 項目 | 内容 |
|------|------|
| ペルソナ名 | スタートアップエンジニアチーム |
| 役割 | 技術リード / 開発メンバー |
| 課題 | AI開発のプロセスが属人的で品質にばらつきがある |
| ゴール | チーム全体でAI開発の品質と速度を標準化する |

### 2.3 ターシャリユーザー: 非エンジニア

| 項目 | 内容 |
|------|------|
| ペルソナ名 | 非エンジニア（AI開発に興味がある人） |
| 役割 | プロダクトオーナー |
| 課題 | 知識がなくてもサービスを作りたいが、AI開発ツールでは思い通りにならない |
| ゴール | 技術知識なしで自分のアイデアをプロダクトにする |

## 3. ユースケース

### 3.1 コアユースケース（必須）

| ID | ユースケース | ユーザー | 優先度 |
|----|------------|---------|-------|
| UC-001 | アイデアからSSOT仕様書一式を自動生成する | 全ユーザー | P0 |
| UC-002 | 仕様書からタスク分解・プロンプト生成・監査を行う | 全ユーザー | P0 |
| UC-003 | リポジトリベースで自動開発を実行する | 全ユーザー | P0 |
| UC-004 | 新規プロジェクトをフレームワークで初期化する | 全ユーザー | P0 |
| UC-005 | 開発進捗をダッシュボードで確認する | 全ユーザー | P1 |
| UC-006 | マーケティング資料を自動生成する | 全ユーザー | P1 |
| UC-007 | 外部ツール（GitHub/Plane/Discord）と連携する | チーム | P1 |

### 3.2 ユーザーストーリー

```
UC-001:
As a 個人開発者,
I want to 「こんなものが作りたい」を対話形式で伝えるだけで仕様書一式が生成される,
so that 仕様策定の専門知識なしにガチガチのSSOT仕様書が手に入る.
```

```
UC-002:
As a 個人開発者,
I want to 仕様書から自動でタスク分解・実装プロンプト・品質監査が行われる,
so that 実装の指示出しや品質管理を自分でやる必要がない.
```

```
UC-003:
As a 個人開発者,
I want to リポジトリベースで自動開発が進み、判断が必要な時だけ確認が来る,
so that PCの前にいなくても開発が進む.
```

```
UC-004:
As a 個人開発者,
I want to 1コマンドでプロジェクトの雛形が整う,
so that セットアップに時間をかけずにすぐ開発を始められる.
```

## 4. スコープ定義

### 4.1 Goals（やること）

- [x] CLIツール: framework init / discover / generate / plan / run / audit / status
- [x] ディスカバリーフロー（5ステージ対話型ヒアリング）の自動化
- [x] SSOT仕様書の自動生成チェーン（ビジネス → プロダクト → 技術）
- [x] タスク分解・実装順序決定の自動化
- [x] 実装プロンプト生成・監査の自動化
- [x] リポジトリベース自動開発（AI実装 + 確認フロー）
- [x] コード監査・テスト監査の自動化
- [ ] Next.jsダッシュボード（進捗・スコア・アラート表示）
- [ ] 外部連携（GitHub Issues/PR、Plane、Discord通知）

### 4.2 Non-Goals（やらないこと）

明示的にスコープ外とするもの：

- IDE プラグイン（Cursor/VS Code拡張）の提供
- ホスティング/インフラの直接管理
- ノーコードビルダー機能（GUIでのアプリ構築）
- マルチテナントSaaS版（v1ではCLIツールのみ）

### 4.3 将来の検討事項（Future Considerations）

今回は対象外だが、将来検討するもの：

- Web版ダッシュボードのSaaS化
- チーム機能（複数人での並行開発管理）
- マーケットプレイス（カスタムテンプレート共有）
- IDE連携プラグイン

## 6. 制約条件

### 6.1 技術制約

- Vercelへのデプロイ
- Vitest によるテスト

| 納期 | MVP: Phase 1（CLIコア機能） |
| 予算 | 個人開発（AI活用） |
| 人員 | 1名 + AI |
| 法規制 | 特になし |

### 6.3 運用制約

- CLIツールはnpmパッケージとして配布
- AI APIキー（Anthropic/OpenAI）はユーザーが用意

## 8. リスク
